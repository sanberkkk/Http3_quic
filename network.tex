\section{Giriş}

Bu rapor, INF 334 yılici projesinde QUIC ve HTTP/3 protokollerinin
pratikte nasıl gözlemlendiğini ve ölçüldüğünü belgeler. Çalışma iki
katmandan oluşur: \textbf{Kısım I} (Docker \texttt{quiche} sunucusu +
multithread Java istemci + loglama + Wireshark) ve \textbf{Bonus}
(\texttt{quiche4j} ile gerçek TLS/HTTP/3 haberleşmesi).

Proje, teorik QUIC/HTTP/3 bilgisini somut kanıtlarla eşleştirir: UDP
üzerinden paket yakalama, olay tabanlı log dosyası, TPS istatistiği ve
(bonus) tam HTTP/3 GET yanıtı.

% ════════════════════════════════════════════════════════════
\section{Teorik Arka Plan}

\subsection{HTTP/2 ve HTTP/3 Karşılaştırması}

HTTP/2, tek TCP bağlantısında çoğullama sağlar; paket kaybında tüm
akışlar etkilenebilir (\textbf{TCP Head-of-Line Blocking}). HTTP/3,
taşıma katmanında UDP tabanlı \textbf{QUIC} kullanır; akışlar birbirinden
bağımsızdır ve TLS 1.3 QUIC'in parçasıdır.

\begin{table}[H]
  \centering
  \caption{HTTP/2 ve HTTP/3 Karşılaştırması}
  \begin{tabular}{lll}
    \toprule
    \textbf{Özellik} & \textbf{HTTP/2} & \textbf{HTTP/3} \\
    \midrule
    Taşıma & TCP & UDP (QUIC) \\
    Güvenlik & TLS 1.2+ & TLS 1.3 (QUIC içinde) \\
    HoL Blocking & Var & Akış bazında yok \\
    Bağlantı kurma & 2--3 RTT & 0--1 RTT \\
  \bottomrule
  \end{tabular}
\end{table}

\subsection{QUIC Protokolü}

QUIC (RFC 9000), el sıkışmayı taşıma katmanına entegre eder; Connection ID
ile bağlantı göçünü destekler. HTTP/3 (RFC 9114) uygulama katmanı olarak
QUIC üzerinde \texttt{h3} ALPN ile çalışır.

\section{Uygulama Mimarisi}
\label{sec:architecture}

\subsection{Sunucu (Kısım I)}

Rust \texttt{quiche} örnek sunucusu Docker içinde \texttt{0.0.0.0:4433}
adresinde UDP dinler; tam QUIC/TLS/HTTP/3 yığını sunucu tarafında Rust ile
çalışır.

\subsection{İstemci: \texttt{QuicheClient} (Kısım I)}

\texttt{part1} modülü PDF'teki \textbf{multithread Java} ve \textbf{loglama}
gereksinimlerini karşılar; her olayda protokol işi Docker'daki
\texttt{http3-client} ile yapılır (tam QUIC/HTTP/3):
\begin{itemize}[leftmargin=1.5em]
  \item \texttt{ExecutorService} -- 5 thread, 10 olay
  \item \texttt{ProcessBuilder} -- \texttt{docker run ... http3-client URL}
  \item \texttt{START} / docker exit 0 $\rightarrow$ \texttt{STOP} / aksi halde \texttt{FAIL}
  \item Hedef (macOS): \texttt{https://host.docker.internal:4433/}
\end{itemize}

\begin{lstlisting}[caption={QuicheClient -- multithread + Docker HTTP/3}]
for (int i = 0; i < NUM_EVENT; i++)
    pool.submit(() -> startEvent(id));
// startEvent -> docker run quiche-proje .../http3-client TARGET_URL
\end{lstlisting}

\begin{infobox}
\textbf{Kısım I vs Bonus:} İkisinde de tam HTTP/3 vardır; Kısım I'de
istemci \textbf{Java + Docker quiche}, bonus'ta \textbf{Java quiche4j (JNI)}
kullanılır.
\end{infobox}

\section{Sonuç ve Değerlendirme}

\begin{itemize}[leftmargin=1.5em]
  \item \textbf{Kısım I} PDF gereksinimlerini karşılar: Docker QUIC sunucusu,
        multithread Java istemci (Docker \texttt{http3-client} ile tam HTTP/3),
        START/STOP/FAIL logları, TPS, Wireshark kanıtı.
  \item \textbf{Bonus} projeyi HTTP/3 seviyesine taşır: Java \texttt{quiche4j}
        ile TLS + \texttt{GET} + \texttt{200 OK} doğrulandı.
  \item İki katmanın karşılaştırması, düşük seviye UDP/NIO ile tam protokol
        yığını arasındaki farkı netleştirir; Wireshark her iki senaryoda
        farklı QUIC aşamalarını görünür kılar.
  \item Gelecek iyileştirme: Kısım I istemcisinde \texttt{quiche4j} entegrasyonu
        veya bonus'ta \texttt{set\_keylog} ile tam Wireshark decrypt.
\end{itemize}
