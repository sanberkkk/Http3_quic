\section{Giriş}

Bu rapor, INF 334 Bilgisayar Ağları dersi kapsamında hazırlanan yılici proje
çalışmasını belgelemektedir. Proje kapsamında Cloudflare'in açık kaynaklı
\texttt{quiche} kütüphanesi (\url{https://github.com/cloudflare/quiche})
Docker ortamında derlenerek çalıştırılmış; QUIC+HTTP/3 tabanlı
sunucu--istemci haberleşmesi gerçeklenmiş; Java ile multithread istemci
yazılmış; Wireshark ile ağ trafiği yakalanmış ve istatistiksel loglar
üretilmiştir.

\section{Teorik Arka Plan}

\subsection{HTTP/2 ve HTTP/3 Karşılaştırması}

HTTP/2, tek bir TCP bağlantısı üzerinden çoğullama (multiplexing) sağlar.
Ancak TCP katmanında yaşanan bir paket kaybı tüm akışları bloklar; buna
\textbf{TCP Head-of-Line Blocking} adı verilir. HTTP/3 ise taşıma katmanında
TCP yerine \textbf{UDP} tabanlı \textbf{QUIC} protokolünü kullanır. Akışlar
birbirinden bağımsız olduğundan bir akıştaki paket kaybı diğerlerini
etkilemez.

\begin{table}[H]
  \centering
  \caption{HTTP/2 ve HTTP/3 Protokollerinin Karşılaştırması}
  \begin{tabular}{lll}
    \toprule
    \textbf{Özellik}          & \textbf{HTTP/2}      & \textbf{HTTP/3}       \\
    \midrule
    Taşıma Katmanı            & TCP                  & UDP (QUIC)            \\
    Güvenlik                  & TLS 1.2+             & TLS 1.3 (QUIC içinde) \\
    Head-of-Line Blocking     & Var                  & Yok                   \\
    Bağlantı Kurma Süresi     & 2--3 RTT             & 0--1 RTT              \\
    Bağlantı Göçü             & Desteklenmez         & 64-bit Connection ID  \\
    \bottomrule
  \end{tabular}
\end{table}

\subsection{QUIC Protokolü}
QUIC, Google tarafından geliştirilen ve IETF tarafından RFC 9000 ile
standardize edilen yeni nesil bir taşıma protokolüdür. Temel özellikleri:
\begin{itemize}
  \item \textbf{0-RTT / 1-RTT bağlantı kurma:} TLS el sıkışması QUIC
        handshake ile birleştirilmiştir.
  \item \textbf{Bağımsız akışlar:} Paket kaybı yalnızca ilgili akışı
        durdurur.
  \item \textbf{Bağlantı göçü:} 64-bit Connection ID sayesinde IP adresi
        değişse dahi (4G $\rightarrow$ Wi-Fi) bağlantı kesilmez.
  \item \textbf{Yerleşik şifreleme:} TLS 1.3 QUIC'in ayrılmaz parçasıdır.
\end{itemize}

\subsection{Docker ile Kurulum}
quiche kütüphanesi Docker ortamında derlenmiştir. Bu yaklaşımın
temel avantajı, farklı işletim sistemi ve mimari farklarından
(Apple Silicon, Intel Mac) bağımsız, tekrar üretilebilir bir
ortam sağlamasıdır. Oluşturulan Docker image ekip üyelerine
dağıtılarak herkesin aynı ortamda çalışması garanti edilmiştir.
\begin{lstlisting}[language=bash, caption={Dockerfile}]
FROM rust:latest
RUN apt-get update && apt-get install -y \
    git cmake pkg-config libssl-dev clang libclang-dev
RUN git clone --recursive \
    https://github.com/cloudflare/quiche
WORKDIR /quiche
RUN cargo build --examples
\end{lstlisting}
\begin{lstlisting}[language=bash, caption={Image oluşturma ve sunucu başlatma}]
# Image olustur
docker build -t quiche-proje .
# Sunucuyu baslat (ornek server, UDP 4433)
docker run -d --name quiche-server -w /quiche/quiche \
  -p 4433:4433/udp quiche-proje \
  /quiche/target/debug/examples/server
# HTTP/3 istemci ile test (host agi)
docker run --rm --network host quiche-proje \
  /quiche/target/debug/examples/http3-client \
  https://cloudflare-quic.com/
\end{lstlisting}

Kurulum tamamlandıktan sonra ortam terminal üzerinden test edilmiştir.
Docker image başarıyla oluşturulduktan sonra iki ayrı terminal sekmesi
açılmıştır. Birinci sekmede \texttt{quiche-server} çalıştırılmış ve
sunucu \texttt{0.0.0.0:4433} adresinde QUIC bağlantılarını beklemeye
başlamıştır. İkinci sekmede ise \texttt{quiche-client} çalıştırılmış

\section{Uygulama Mimarisi}
\subsection{Sunucu Tarafı}
Sunucu tarafı, quiche kütüphanesinin örnek sunucusu
\texttt{quiche-server} kullanılarak Docker container içinde
çalıştırılmıştır. Sunucu \texttt{0.0.0.0:4433} adresinde UDP
üzerinden QUIC bağlantılarını dinlemektedir. İstemciden alınan
\texttt{Not Found!} yanıtı, QUIC el sıkışmasının ve HTTP/3
iletişiminin başarıyla gerçekleştiğini doğrulamaktadır.
\subsection{İstemci Tarafı -- Multithread Java (\texttt{QuicheClient})}

Kısım I'de \texttt{part1} modülündeki \texttt{QuicheClient.java} kullanılmıştır.
İstemci, \texttt{java.nio.channels.DatagramChannel} ile UDP üzerinden QUIC
\textbf{Initial} formatında sabit boyutlu paketler gönderir; \texttt{ExecutorService}
ile eşzamanlı görevler çalıştırılır. Her olay için \texttt{START}, başarılı
gönderimde \texttt{STOP}, hata durumunda \texttt{FAIL} kaydı üretilir.

\begin{lstlisting}[caption={QuicheClient.java -- çekirdek yapı}]
ExecutorService executorService =
    Executors.newFixedThreadPool(THREAD_COUNT);
logger = new HighPerformanceLogger(
    Paths.get("quic_project_logs.txt"), 10_000,
    HighPerformanceLogger.OverflowPolicy.BLOCK);
tracker = new StatisticsTracker();
for (int i = 0; i < NUM_EVENT; i++) {
    final int id = i;
    executorService.submit(() -> startEvent(id));
}
// startEvent: DatagramChannel -> 127.0.0.1:4433
// QUIC Initial header (26 byte) + payload
\end{lstlisting}

\textbf{Not:} Bu istemci tam TLS/HTTP/3 el sıkışması yapmaz; Docker
\texttt{examples/server} ile birlikte Wireshark'ta Initial paketleri ve
loglama/istatistik gereksinimleri karşılanır. Gerçek HTTP/3 bonus
bölümünde quiche4j ile sağlanmıştır.

\subsection{Loglama ve İstatistik Modülü (\texttt{part3})}

\texttt{com.networkproject.logger} paketi: \texttt{HighPerformanceLogger}
(kilitlenmesiz kuyruk), \texttt{StatisticsTracker} (TPS), \texttt{LogEvent}
(\texttt{START}/\texttt{STOP}/\texttt{FAIL}). Çıktı dosyası:
\texttt{quic\_project\_logs.txt}.

\section{Bonus: quiche4j Analizi}

\subsection{quiche (Rust) vs quiche4j (Java) Karşılaştırması}

\texttt{quiche4j}, Rust ile yazılmış \texttt{quiche} kütüphanesinin Java üzerinden
\textbf{JNI (Java Native Interface)} aracılığıyla kullanılmasını sağlayan bir
bağlayıcı katmandır.

\begin{table}[H]
  \centering
  \caption{quiche (Rust) ve quiche4j (Java) Karşılaştırması}
  \begin{tabular}{lll}
    \toprule
    \textbf{Özellik}       & \textbf{quiche (Rust)}         & \textbf{quiche4j (Java)} \\
    \midrule
    Dil                    & Rust                           & Java (JNI üzerinden)    \\
    Performans             & Çok yüksek                     & Yüksek                  \\
    Bellek güvenliği       & Derleyici garantisi            & JVM GC                  \\
    API düzeyi             & Düşük seviye                   & Düşük seviye            \\
    Ekosistem entegrasyonu & Cargo                          & Maven / Gradle          \\
    Kullanım kolaylığı     & Orta                           & Java geliştirici dostu  \\
    \bottomrule
  \end{tabular}
\end{table}

\subsection{JNI Kurulumu ve Çalıştırma}

Bonus için \texttt{quiche4j} deposu klonlanmış, \texttt{quiche4j-jni}
release modda derlenmiş ve \texttt{QUICHE\_JNI\_LIB} ile Java'ya
yüklenmiştir (\texttt{bonus/setup-jni.sh}).

\begin{lstlisting}[language=bash, caption={Bonus sunucu ve istemci (iki terminal)}]
cd /Users/sanberk96/Projects/network
export QUICHE_JNI_LIB=/tmp/quiche4j-native/release
export SSLKEYLOGFILE=logs/sslkeys-bonus.log
./bonus/run-server.sh 4433
# Baska terminal:
./bonus/run-client.sh https://127.0.0.1:4433/
\end{lstlisting}

\textbf{Önemli:} Bonus sunucusu çalışırken Docker \texttt{quiche-server}
aynı 4433 portunu kullanmamalıdır (\texttt{docker stop quiche-server}).

\subsubsection{Wireshark -- Bonus (şifre çözülmüş QUIC)}

\begin{enumerate}[leftmargin=1.5em]
  \item Wireshark: arayüz \textbf{Loopback: lo0} (veya Any), capture başlat.
  \item TLS $\rightarrow$ (Pre)-Master-Secret log dosyası:
        \texttt{logs/sslkeys-bonus.log} (her iki terminalde
        \texttt{SSLKEYLOGFILE} export edilmeli).
  \item Sunucu + istemciyi çalıştır; filtre: \texttt{udp.port == 4433}.
  \item Paket ayrıntısında \textbf{Decrypted QUIC}, ALPN \texttt{h3-29},
        HTTP/3 pseudo-header (\texttt{:method: GET}, \texttt{:status: 200}).
\end{enumerate}

\section{Sonuç ve Değerlendirme}

Bu proje kapsamında QUIC/HTTP/3 kavramları Docker \texttt{quiche} ile,
multithread Java istemci ve loglama modülü ile, bonus kısımda ise
\texttt{quiche4j} (JNI) ile uçtan uca doğrulanmıştır. Elde edilen başlıca
çıkarımlar:

\begin{itemize}[leftmargin=1.5em]
  \item HTTP/3, UDP üzerinde QUIC ile TCP head-of-line blocking'i azaltır.
  \item Kısım I: \texttt{DatagramChannel} + loglama ile ölçülebilir TPS;
        Wireshark'ta Initial paketleri.
  \item Bonus: Gerçek HTTP/3 GET ve \texttt{200 OK}; Rust quiche ile aynı
        protokol yığını, Java API üzerinden.
  \item Wireshark, yerel loopback ve dış Cloudflare trafiğinde QUIC
        el sıkışmasını görselleştirmek için kritik araçtır.
\end{itemize}
